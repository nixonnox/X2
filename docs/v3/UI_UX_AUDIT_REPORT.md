## 감사 요약

본 보고서는 X2 플랫폼의 프론트엔드 UI/UX를 대상으로 실시한 QA/QC 감사 결과를 기술한다. 총 **12건**의 이슈가 식별되었으며, S0 블로커 1건, S1 크리티컬 3건, S2 상위 5건, S3 중위 3건으로 분류되었다. 현재 단계에서 프로덕션 배포 전 반드시 S0/S1 이슈를 해결해야 한다.

## 이슈 상세

### UXA-001 | S0 블로커 | 데모 데이터 표시 시 시스템 수준 경고 미비

- **화면/컴포넌트**: 대시보드 전체 (dashboard, comments, competitors, channel detail, insights, reports)
- **관련 파일**:
  - `apps/web/src/components/shared/demo-banner.tsx` (DemoBanner 컴포넌트)
  - `apps/web/src/app/(dashboard)/dashboard/page.tsx` (배너 적용됨)
  - `apps/web/src/app/(dashboard)/comments/page.tsx` (배너 적용됨)
  - `apps/web/src/app/(dashboard)/competitors/page.tsx` (배너 적용됨)
- **설명**: `DemoBanner` 컴포넌트가 구현되어 3개 페이지(dashboard, comments, competitors)에 추가되었으나, 채널 상세(`/channels/[id]`), 인사이트(`/insights`), 리포트(`/reports`) 등 다른 mock-data 화면에는 배너가 적용되지 않았다. 사용자가 데모 데이터를 실제 분석 결과로 오인할 위험이 있다.
- **사용자 영향**: 사용자가 mock 데이터 기반 수치를 실제 소셜 미디어 분석 결과로 착각하여 잘못된 의사결정을 내릴 수 있다.
- **스크린샷 참조**: 해당 페이지에서 DemoBanner 유무 비교 스크린샷 필요
- **수정 권고**:
  1. mock 데이터를 사용하는 모든 페이지에 `DemoBanner`를 추가하거나, 레이아웃 수준에서 전역 배너를 렌더링한다.
  2. 환경 변수 또는 feature flag(`NEXT_PUBLIC_DEMO_MODE`)를 기반으로 레이아웃(`apps/web/src/app/(dashboard)/layout.tsx`)에서 일괄 표시하는 방식을 권장한다.
  3. 배너 메시지를 더 강조하여(빨간색 경고 또는 상단 고정 바) 사용자가 놓치지 않도록 한다.

### UXA-002 | S1 크리티컬 | 온보딩 위저드 API 미연결

- **화면/컴포넌트**: 온보딩 (`/onboarding`)
- **관련 파일**:
  - `apps/web/src/app/(auth)/onboarding/page.tsx`
  - `apps/web/src/app/(auth)/onboarding/onboarding-wizard.tsx`
- **설명**: 온보딩 위저드가 `setTimeout(r, 500)` 플레이스홀더를 사용하여 API 호출을 시뮬레이션한다. 워크스페이스 생성(1단계)과 프로젝트 생성(2단계) 모두 `// TODO: 실제 API 호출 (tRPC)` 주석이 남아 있으며 실제 DB에 아무것도 생성되지 않는다.
- **사용자 영향**: 사용자가 온보딩을 완료했다고 인지하지만 실제로는 워크스페이스/프로젝트/채널이 생성되지 않아, 이후 대시보드에서 빈 상태만 표시되거나 오류가 발생한다.
- **스크린샷 참조**: 온보딩 완료 후 대시보드 빈 상태 스크린샷
- **수정 권고**:
  1. tRPC `workspace.create` 및 `project.create` 뮤테이션을 연결한다.
  2. 온보딩 완료 후 실제 데이터 존재 여부를 확인하는 검증 로직을 추가한다.
  3. API 호출 실패 시 사용자에게 명확한 에러 메시지를 표시한다.

### UXA-003 | S1 크리티컬 | Evidence Bundle UI 구현률 약 5%

- **화면/컴포넌트**: 인사이트 > Evidence (`/insights/evidence`)
- **관련 파일**:
  - `docs/v3/EVIDENCE_BUNDLE_DESIGN.md` (8종 번들 타입, 전체 설계)
- **설명**: `EVIDENCE_BUNDLE_DESIGN.md`에서 스크린샷 캡처, 타임라인 뷰, PDF/PPT 내보내기, 8종 번들 타입(SENTIMENT_OVERVIEW, FAQ_SUMMARY, RISK_SUMMARY 등)을 포함한 풍부한 기능을 설계했으나, 실제 코드에는 빈 상태 컴포넌트만 존재한다. Evidence Bundle 관련 컴포넌트 파일이 검색되지 않는다.
- **사용자 영향**: 리포트 근거 자료 기능이 작동하지 않아, 사용자가 분석 결과를 PPT/문서에 활용할 수 없다.
- **스크린샷 참조**: Evidence 탭 빈 상태 화면
- **수정 권고**:
  1. 최소 MVP로 SENTIMENT_OVERVIEW 및 TREND_OVERVIEW 번들 렌더링을 우선 구현한다.
  2. 나머지 6종은 Phase별로 점진 추가한다.
  3. 미구현 번들 타입에는 "준비 중" 표시를 명확히 한다.

### UXA-004 | S1 크리티컬 | 이메일 기능 비활성화 상태에서 UI 잔존

- **화면/컴포넌트**: 리포트 상세, 리포트 생성
- **관련 파일**:
  - `apps/web/src/app/(dashboard)/insights/reports/[id]/page.tsx`
  - `apps/web/src/app/(dashboard)/insights/reports/new/page.tsx`
- **설명**: 이메일 전송 버튼이 "준비 중" 라벨로 비활성화되어 있으나, 이메일 관련 UI 요소(입력 필드, 수신자 목록 등)가 다른 위치에 여전히 노출될 수 있다. 비활성화된 기능의 UI가 부분적으로 보이는 것은 사용자 혼란을 야기한다.
- **사용자 영향**: 사용자가 이메일 전송을 시도하다 실패하거나, "준비 중" 상태가 언제 해제되는지 알 수 없어 기능 신뢰도가 하락한다.
- **스크린샷 참조**: 이메일 전송 버튼 disabled 상태 스크린샷
- **수정 권고**:
  1. 이메일 기능을 feature flag로 완전히 숨기거나, 명확한 출시 예정 안내를 제공한다.
  2. "준비 중" 대신 구체적인 안내 ("이메일 전송 기능은 Pro 플랜 이상에서 제공 예정입니다") 를 표시한다.
  3. disabled 버튼에 tooltip으로 사유를 표시한다.

### UXA-005 | S2 상위 | 플랜 기반 UI 게이팅 부재

- **화면/컴포넌트**: 전체 애플리케이션
- **설명**: FREE, PRO, BUSINESS 플랜 구분이 존재하지만 UI에서 플랜별 기능 제한이 전혀 구현되지 않았다. 모든 기능이 모든 플랜 사용자에게 동일하게 노출된다. 업그레이드 프롬프트, 기능 잠금 UI, 사용량 제한 안내가 없다.
- **사용자 영향**: 유료 전환 동기가 없으며, 무료 사용자가 유료 기능을 사용하려다 백엔드에서 거부당할 때 UX가 불분명하다.
- **수정 권고**:
  1. `useSubscription` 훅을 생성하여 현재 플랜 정보를 조회한다.
  2. 제한 기능에 잠금 아이콘 + "Pro 전용" 배지를 표시한다.
  3. 기능 접근 시 업그레이드 모달을 렌더링한다.

### UXA-006 | S2 상위 | 역할 기반 뷰 제한 부재

- **화면/컴포넌트**: 전체 애플리케이션
- **관련 파일**:
  - `docs/v3/ROLE_BASED_OUTPUT_STRATEGY.md` (역할별 제한 설계)
- **설명**: `ROLE_BASED_OUTPUT_STRATEGY.md`에서 OWNER, ADMIN, MEMBER, VIEWER 역할별 데이터 접근 및 UI 노출 범위를 상세히 정의했으나, 프론트엔드에서 역할 기반 뷰 제한이 전혀 구현되지 않았다. 모든 사용자가 동일한 데이터와 기능에 접근할 수 있다.
- **사용자 영향**: VIEWER 역할 사용자가 설정 변경이나 데이터 삭제 등 권한 밖 작업의 UI를 볼 수 있어, 보안 및 거버넌스 문제가 발생할 수 있다.
- **수정 권고**:
  1. `useRole` 훅 또는 RBAC 컨텍스트 프로바이더를 구현한다.
  2. 역할별로 사이드바 메뉴 항목, 버튼, 데이터 컬럼을 조건부 렌더링한다.
  3. 백엔드 API 레벨 권한 검사와 프론트엔드 UI 제한을 동기화한다.

### UXA-009 | S2 상위 | DataStatusBar 실제 데이터 미연결

- **화면/컴포넌트**: 대시보드 상단 (`/dashboard`)
- **관련 파일**:
  - `apps/web/src/app/(dashboard)/dashboard/dashboard-view.tsx`
- **설명**: `DataStatusBar` 컴포넌트가 대시보드에 존재하지만 실제 컬렉션 파이프라인 상태(마지막 수집 시각, 수집 성공률, 큐 대기 건수 등)와 연결되지 않았다. 정적 또는 mock 값을 표시하고 있다.

  1. tRPC `collection.status` 엔드포인트와 연결하여 실시간 상태를 표시한다.
  2. 수집 지연 시 경고 아이콘과 "마지막 수집: X분 전" 표시를 추가한다.

### UXA-010 | S3 중위 | i18n 커버리지 불명확

- **화면/컴포넌트**: 전체 애플리케이션
- **관련 파일**:
  - `apps/web/src/messages/ko.json`
  - `apps/web/src/messages/en.json`
- **설명**: `next-intl`이 구성되어 있고 `useTranslations` 사용이 다수 파일(sidebar, dashboard, comments, competitors, insights 등 15개 이상)에서 확인되나, 하드코딩된 한국어 문자열과 번역 키 사용이 혼재되어 있다. 영문 메시지 파일의 완성도가 불명확하다.
- **사용자 영향**: 영어 사용자에게 한국어가 혼재되어 표시되거나, 번역 키가 그대로 노출될 수 있다.
- **수정 권고**:
  1. 하드코딩된 한국어 문자열을 전수 조사하여 번역 키로 전환한다.
  2. `ko.json`과 `en.json`의 키 동기화 검사 스크립트를 CI에 추가한다.
  3. Korean-first 정책에 따라 `ko.json`을 기준 파일로 관리한다.

### UXA-011 | S3 중위 | 로딩/에러 상태 일관성 부족

- **화면/컴포넌트**: 전체 페이지
- **설명**: 일부 페이지(dashboard, channels)는 `Suspense` 경계와 로딩 스켈레톤이 적용되어 있으나, 다른 페이지에서는 로딩/에러 상태 처리가 누락되었거나 일관되지 않은 패턴을 사용한다.
- **사용자 영향**: 데이터 로딩 중 빈 화면이 표시되거나, API 오류 시 사용자가 대응할 수 없는 상태에 놓인다.
- **수정 권고**:
  1. `loading.tsx` 및 `error.tsx` 파일을 모든 라우트 세그먼트에 추가한다.
  2. 공통 로딩 스켈레톤 컴포넌트와 에러 바운더리를 표준화한다.
  3. tRPC 쿼리에 대한 일관된 에러 핸들링 패턴을 적용한다.

### UXA-012 | S3 중위 | 모바일 반응형 미감사

- **화면/컴포넌트**: 전체 애플리케이션
- **설명**: Tailwind CSS 반응형 유틸리티를 사용하고 있으나, 모바일 환경에서의 체계적인 테스트가 수행되지 않았다. 사이드바 모바일 대응, 대시보드 카드 레이아웃, 테이블 가로 스크롤 등의 반응형 동작이 검증되지 않았다.
- **사용자 영향**: 모바일 사용자가 레이아웃 깨짐, 터치 영역 부족, 가로 스크롤 불편 등을 경험할 수 있다.
- **수정 권고**:
  1. 주요 브레이크포인트(sm: 640px, md: 768px, lg: 1024px)에서 전체 페이지를 검증한다.
  2. 사이드바의 모바일 드로어 전환을 확인한다.
  3. 데이터 테이블에 가로 스크롤 또는 카드형 레이아웃 대체를 적용한다.
