# ListeningMind 구현 우선순위

> 작성일: 2026-03-13
> 목적: 구현 순서, 최대 blocker, 단계별 로드맵 제시

---

## 현재 X2 구조에서 가장 큰 Blocker

### Blocker #1: 실제 데이터가 없다 (심각도: 최상)

현재 X2의 모든 분석은 **시뮬레이션 데이터**로 동작합니다.

- 검색량: 수식으로 만든 가짜 숫자
- 트렌드: 랜덤 함수로 생성된 12개월 데이터
- 연관 키워드: 템플릿(접미사 조합)으로 만든 가짜 키워드
- 소셜 볼륨: 랜덤 함수로 생성

**결과적으로 어떤 분석을 하든 "실제 소비자 행동"이 반영되지 않습니다.**
이것이 해결되지 않으면 나머지 모든 기능 개선은 의미가 제한적입니다.

### Blocker #2: SERP 데이터 수집 체계가 없다 (심각도: 상)

리스닝마인드의 핵심 경쟁력은 **SERP(검색 결과 페이지) 분석**입니다.

- 검색 경로: SERP의 연관 검색어/추천 검색어에서 추출
- 클러스터링: SERP 결과 공유도 기반
- 도메인 분석: SERP 상위 10개 URL 분석

X2에는 SERP를 수집하는 체계가 전혀 없습니다.

### Blocker #3: DB 마이그레이션 미적용 (심각도: 중)

5개 새 테이블의 스키마는 작성되었지만, PostgreSQL이 꺼져 있어 실제 DB에 적용되지 않았습니다.

### Blocker #4: 이중 분석 시스템 (심각도: 중)

REST API(`/api/intent/analyze`)와 tRPC(`routers/intent.ts`)가 독립적으로 존재합니다.
프론트엔드는 REST만 사용하고, tRPC는 호출되지 않습니다.
하나로 통일해야 유지보수가 가능합니다.

## 구현 우선순위

### Phase 1: 데이터 기반 확보 (반드시 먼저)

> 목표: "가짜 데이터"를 "진짜 데이터"로 교체

| #   | 작업                         | 설명                                         | 난이도 | 소요  |
| --- | ---------------------------- | -------------------------------------------- | ------ | ----- |
| 1-1 | **Google Ads API 연동**      | 실제 월간 검색량, CPC, 광고 경쟁도 수집      | 중     | 3-5일 |
| 1-2 | **Naver 검색 API 연동**      | Naver 검색량, 자동완성 수집                  | 중     | 2-3일 |
| 1-3 | **Google Autocomplete 연동** | 실제 자동완성 추천 검색어 수집               | 하     | 1-2일 |
| 1-4 | **Google Trends 연동**       | 트렌드 시계열 데이터 (pytrends 또는 SerpAPI) | 중     | 2-3일 |
| 1-5 | **DB 마이그레이션 적용**     | PostgreSQL 시작 + 스키마 적용                | 하     | 0.5일 |
| 1-6 | **분석 시스템 통일**         | REST/tRPC 이중 구조 → 하나로 정리            | 중     | 2-3일 |

**Phase 1 완료 시 반영률: 25% → ~45%**

왜 이것이 먼저인가:

- 실제 데이터 없이는 모든 분석/시각화가 "데모"에 불과
- 나머지 모든 기능이 이 데이터에 의존
- API 연동은 독립적으로 진행 가능 (프론트엔드 변경 최소)

---

### Phase 2: 핵심 UX 개선 + 인텐트 파인더 완성

> 목표: 가장 기본이 되는 인텐트 파인더를 리스닝마인드 수준으로 끌어올림

| #   | 작업                         | 설명                                         | 난이도 | 소요  |
| --- | ---------------------------- | -------------------------------------------- | ------ | ----- |
| 2-1 | **복수 키워드 입력**         | 태그 입력 UI (최대 100개)                    | 하     | 1일   |
| 2-2 | **트렌드 스파크라인**        | 키워드 테이블에 미니 라인 차트               | 하     | 1일   |
| 2-3 | **테이블 정렬/페이지네이션** | 검색량/CPC/갭점수 기준 정렬 + 페이지네이션   | 하     | 1일   |
| 2-4 | **CSV 다운로드**             | 키워드 테이블 내보내기                       | 하     | 0.5일 |
| 2-5 | **검색 의도 4분류**          | I/N/C/T 분류 체계 추가 (기존 5분류와 병행)   | 중     | 2일   |
| 2-6 | **분석 히스토리 DB 저장**    | 분석 결과 영속화 + 재열람                    | 중     | 2일   |
| 2-7 | **기능 간 연결**             | 키워드 클릭 → 패스파인더/클러스터파인더 이동 | 하     | 1일   |

**Phase 2 완료 시 반영률: ~45% → ~55%**

---

### Phase 3: SERP 기반 엔진 구축

> 목표: 리스닝마인드의 핵심 차별점인 SERP 기반 분석 체계 구축

| #   | 작업                     | 설명                                                         | 난이도 | 소요  |
| --- | ------------------------ | ------------------------------------------------------------ | ------ | ----- |
| 3-1 | **SERP 수집기**          | SerpAPI 또는 DataForSEO 연동 (상위 10 URL, 연관 검색어, PAA) | 상     | 5-7일 |
| 3-2 | **SERP 캐시/저장**       | SerpResult 테이블 + 캐시 정책                                | 중     | 2일   |
| 3-3 | **검색 경로 빌더**       | SERP 연관 검색어 기반 before→target→after 그래프 구축        | 상     | 5-7일 |
| 3-4 | **SERP 기반 클러스터링** | SERP 결과 공유도 기반 키워드 그룹핑                          | 상     | 5-7일 |
| 3-5 | **도메인 경쟁 분석**     | SERP에서 도메인별 노출 점유율 계산                           | 중     | 3일   |

**Phase 3 완료 시 반영률: ~55% → ~75%**

왜 Phase 3가 Phase 2 다음인가:

- SERP 수집은 비용이 발생하는 외부 API 의존 (SerpAPI: $50~200/월)
- Phase 1~2가 완성된 상태에서 데이터 품질을 검증한 후 진행하는 것이 안전
- SERP 기반 기능은 다른 기능에 비해 구현 복잡도가 높음

## 핵심 의사결정 필요 사항

| 결정 사항                 | 선택지                                    | 영향                           |
| ------------------------- | ----------------------------------------- | ------------------------------ |
| **SERP 데이터 소스**      | SerpAPI vs DataForSEO vs 자체 크롤링      | 비용, 속도, 안정성             |
| **Google Ads API 계정**   | 기존 계정 사용 vs 새 계정                 | 데이터 정확도 (광고 지출 여부) |
| **분석 시스템 통일**      | REST 유지 vs tRPC 전환                    | 코드 구조, 유지보수            |
| **키워드 규모 목표**      | 1,000개 vs 5,000개 vs 20,000개            | 인프라 비용, SERP API 호출 수  |
| **저니 파인더 포함 여부** | 6대 기능만 vs 7대 기능 (저니 파인더 추가) | 개발 일정                      |
