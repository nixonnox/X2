# Notification Count Source of Truth

## Overview

This document defines the single source of truth for notification unread counts
across the entire application. Both the top-bar Bell badge and the Dashboard
IntelligenceSummaryCard now derive their unread count from the same tRPC query,
eliminating the previous inconsistency where two different counting strategies
could diverge.

---

## Source of Truth

| Item              | Value                                                        |
| ----------------- | ------------------------------------------------------------ |
| tRPC query        | `notification.unreadCount`                                   |
| Database table    | `Notification`                                               |
| Filter condition  | `WHERE isRead = false AND userId = :currentUserId`           |
| Return type       | `{ count: number }`                                          |
| Polling interval  | 30 seconds                                                   |

The `notification.unreadCount` query is the **only** sanctioned way to obtain
the number of unread notifications. Any component that needs to display or
reason about unread notification volume MUST call this query rather than
computing the count locally.

---

## Before: Signal Hint String Matching

The Dashboard's IntelligenceSummaryCard previously attempted to derive an
"alert count" by inspecting the `hint` field on live signal entries and
matching specific string patterns (e.g., checking whether the hint text
contained keywords like "급등" or "이상"). This approach had several
fundamental problems:

1. **Fragile coupling** -- The card's count depended on the exact wording of
   hint strings. Any change to hint generation logic silently broke the count.
2. **Inconsistency with Bell** -- The Bell badge read from the Notification
   table while the Dashboard card read from signal hints, so the two numbers
   could diverge even when viewing the same data.
3. **No read-state tracking** -- Signal hints have no concept of "read" or
   "unread". The Dashboard card always showed the total number of matching
   hints, not the number the user had not yet seen.
4. **Race conditions** -- Signal hints could update mid-session as new
   collection cycles completed, causing the count to jump without any
   corresponding notification being created.
5. **Missing coverage** -- Notifications generated by non-signal sources
   (e.g., benchmark threshold alerts, scheduled report completions) never
   appeared in the hint-based count.

---

## After: Unified Query

Both the Bell badge (rendered in the top-bar layout) and the Dashboard
IntelligenceSummaryCard now call `notification.unreadCount` with the same
parameters. The flow is:

```
Bell component
  └─ useQuery("notification.unreadCount", { refetchInterval: 30_000 })
       └─ reads Notification table WHERE isRead = false

Dashboard IntelligenceSummaryCard
  └─ useQuery("notification.unreadCount", { refetchInterval: 30_000 })
       └─ reads Notification table WHERE isRead = false
```

Because both consumers hit the same database query, their displayed counts
are guaranteed to be identical at any given polling tick.

---

## Polling Strategy

Both the Bell component and the Dashboard card use a 30-second polling
interval configured via `refetchInterval: 30_000` in the tRPC `useQuery`
options. This value was chosen as a balance between responsiveness and
server load.

### Polling Lifecycle

| Event                        | Effect                                              |
| ---------------------------- | --------------------------------------------------- |
| Component mounts             | Initial fetch fires immediately                     |
| Every 30 seconds             | Background refetch (no loading spinner)              |
| Tab becomes visible          | Refetch fires if stale (refetchOnWindowFocus: true)  |
| Tab hidden                   | Polling pauses automatically (React Query default)   |
| markRead mutation succeeds   | Explicit invalidation triggers immediate refetch     |
| markAllRead mutation succeeds| Explicit invalidation triggers immediate refetch     |

### Why 30 Seconds

- Intelligence alert generation is itself batched on a schedule (typically
  every 5-15 minutes). Polling more frequently than 30 seconds would not
  surface alerts meaningfully faster.
- At 30 seconds, a page with both Bell and Dashboard card produces at most
  4 lightweight COUNT queries per minute -- negligible server impact.
- Users performing active investigation (clicking into alerts) trigger
  immediate refetches via mutation callbacks, so the 30-second floor only
  applies to passive observation.

---

## Sync Guarantee

The synchronization guarantee rests on three pillars:

1. **Single table** -- All notification state lives in the `Notification`
   table. There is no secondary cache, no in-memory store, and no derived
   materialized view.
2. **Single query** -- Both consumers call the same `notification.unreadCount`
   router procedure. There is no alternative endpoint that returns a count.
3. **Mutation-triggered invalidation** -- The `markRead` and `markAllRead`
   mutations both call `utils.notification.unreadCount.invalidate()` and
   `utils.notification.list.invalidate()` in their `onSuccess` callbacks.
   This ensures that after any read-state change, both the Bell and the
   Dashboard card refetch immediately rather than waiting for the next
   polling tick.

### Invalidation Flow

```
User clicks "모두 읽음" in Bell dropdown
  └─ markAllRead.mutate()
       └─ server: UPDATE Notification SET isRead = true WHERE userId = :id
       └─ onSuccess:
            ├─ utils.notification.unreadCount.invalidate()  // Bell + Dashboard
            └─ utils.notification.list.invalidate()         // Bell dropdown list
```

Both the Bell badge and the Dashboard card subscribe to the same query key,
so a single `invalidate()` call causes both to refetch.

---

## Mutation Reference

| Mutation                      | Server Effect                                       | Client Invalidation                        |
| ----------------------------- | --------------------------------------------------- | ------------------------------------------ |
| `notification.markRead`       | Sets `isRead = true` on one Notification row        | `unreadCount` + `list`                     |
| `notification.markAllRead`    | Sets `isRead = true` on all rows for current user   | `unreadCount` + `list`                     |

Both mutations accept standard tRPC error handling. On failure, the count
is not modified and the UI retains its previous value until the next
polling cycle naturally corrects it.

## Dashboard Card Integration Detail

The IntelligenceSummaryCard previously rendered a count badge derived from
signal hints. After the migration:

- The card imports `api.notification.unreadCount.useQuery`.
- It renders the returned `count` value directly in its badge.
- When `count` is 0, the badge is hidden.
- The card no longer imports or references any signal-hint utilities.
- The card's "알림 보기" link navigates to the Bell dropdown (or, when a
  dedicated `/notifications` page is built, will link there instead).

## Future: WebSocket Real-Time Updates

When WebSocket support is added to the notification subsystem, both the
Bell badge and the Dashboard card will benefit automatically:

1. The server will emit a `notification:countChanged` event whenever a
   new Notification row is inserted or an existing row's `isRead` field
   is updated.
2. A shared React hook (e.g., `useNotificationSocket`) will listen for
   this event and call `utils.notification.unreadCount.invalidate()`.
3. Because both Bell and Dashboard already subscribe to the same query
   key, the invalidation will cause both to refetch immediately.
4. The 30-second polling interval will remain as a fallback in case the
   WebSocket connection drops or reconnects.

No component-level changes will be required. The migration path is:

- Add WebSocket transport layer.
- Add the shared invalidation hook.
- Mount the hook in the app layout (once).
- Both Bell and Dashboard update in real time.
