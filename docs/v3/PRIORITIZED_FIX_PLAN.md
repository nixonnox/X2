## 구현 순서 및 근거

### 권장 구현 순서

```
1. FIX-001 (S)  ← 모든 것의 전제 조건. 가장 먼저, 반드시
2. FIX-004 (S)  ← 001과 동시 진행 가능. Mock 화면 표시 개선
3. FIX-008 (S)  ← 001과 동시 진행 가능. 보안 가드
4. FIX-003 (M)  ← 001 완료 직후. 사용자 첫 진입점
5. FIX-002 (XL) ← 001 완료 직후 시작. 가장 큰 작업
6. FIX-005 (M)  ← 002와 병행 가능. 독립적
7. FIX-007 (M)  ← 002와 병행 가능. 독립적
8. FIX-006 (L)  ← 005 이후 권장 (알림 전달이 가능해야 큐 의미 있음)
9. FIX-009 (M)  ← 002 완료 후
10. FIX-010 (M) ← 009와 병행
11~15. S2 항목들 ← GA 전
16~22. S3 항목들 ← GA 이후
```

### 근거

1. **FIX-001이 최우선인 이유**: 47개 서비스 중 1개만 접근 가능한 상태는 플랫폼이 사실상 비작동 상태임을 의미. 이 수정 없이는 다른 어떤 프론트엔드-백엔드 연결 작업도 불가능.

2. **FIX-003이 FIX-002보다 먼저인 이유**: 온보딩은 사용자 여정의 시작점. 워크스페이스가 생성되지 않으면 나머지 기능이 무의미. 또한 공수가 M으로 002(XL)보다 빠르게 완료 가능.

3. **FIX-002가 단독 스프린트인 이유**: 6개 mock 파일, 8개 소비 파일을 전환하는 대규모 작업. 각 페이지마다 로딩/에러/빈 상태 UI 설계가 필요하여 집중 투입이 효율적.

4. **FIX-005, 007이 006보다 먼저인 이유**: 이메일 전달과 보고서 내러티브는 큐 없이도 동기적으로 작동 가능. BullMQ 셋업(Redis 인프라 포함)은 더 큰 인프라 변경이므로 이후에 진행.

5. **FIX-015(테스트)가 후순위인 이유**: 현재 코드 구조가 FIX-001~002로 크게 변경될 예정. 안정화 단계에서 테스트를 작성하는 것이 재작업을 줄임.
