# 전역 규칙 vs 프로젝트 예외 분리 기준

> 어떤 규칙이 전역이고, 어떤 규칙이 프로젝트별 예외인지 판단하는 기준.

---

## A. 전역 공통 규칙 (모든 프로젝트)

이 규칙은 프로젝트 성격에 관계없이 **항상** 적용됩니다.

| 영역 | 규칙 | 파일 |
|------|------|------|
| 화면 구조 | 한 화면 = 한 핵심 행동 | ux-global-principles.md |
| CTA | 주 CTA 1개, 보조 1~2개, 위험 0~1개 | ux-global-principles.md |
| 정보 계층 | 핵심→이유→근거→세부 | ux-global-principles.md |
| 선택 방식 | 설명 필요한 선택 = 카드형 | decision-ui-global-guide.md |
| 상태 UX | 모든 상태에 설명+이유+행동 | empty-state-global-library.md |
| Writing 톤 | 해요체, 능동형, 긍정형 | ux-writing-global-guide.md |
| Writing 금지 | "불가", "오류 발생", 사용자 탓 표현 | ux-writing-global-guide.md |
| 버튼 문구 | 기능명이 아닌 행동 중심 | ux-writing-global-guide.md |
| 용어 | 기술 용어를 사용자 언어로 변환 | ux-global-principles.md |
| Empty state | 제목+설명+CTA 구조 | empty-state-global-library.md |

---

## B. 프로젝트별 가변 규칙

이 규칙은 프로젝트 성격에 따라 **다르게** 적용할 수 있습니다.

| 영역 | 가변 요소 | 예시 |
|------|----------|------|
| **톤 강도** | 캐주얼 ↔ 포멀 | 금융 앱은 더 포멀, 게임 도구는 더 캐주얼 |
| **유머 강도** | 없음 ↔ 약한 재치 | 의료 앱은 유머 금지, 커뮤니티 도구는 허용 |
| **도메인 용어** | 업종별 전문 용어 사전 | "스킨케어", "포트폴리오", "MAU" 등 |
| **규제 문구** | 법적 필수 문구 | 금융: "원금 손실 가능", 의료: "의사 상담 필요" |
| **시각화 우선순위** | 차트 vs 테이블 vs 카드 | 데이터 도구는 테이블 우선, 대시보드는 차트 우선 |
| **경어 수준** | 해요체 vs 합니다체 | B2B 엔터프라이즈는 합니다체 고려 |
| **언어** | 한국어 vs 영어 vs 다국어 | 글로벌 제품은 i18n 고려 |
| **CTA 문구** | 제품 고유 행동 | "분석 시작" vs "포스팅" vs "주문하기" |

---

## 분리 판단 기준

새 규칙을 추가할 때 아래 질문으로 전역/프로젝트를 판단합니다:

### 전역으로 가는 경우
- [ ] 이 규칙은 제품 도메인과 무관한가?
- [ ] 다른 프로젝트에서도 똑같이 적용해야 하는가?
- [ ] 사용자 경험의 **구조적** 원칙인가?

### 프로젝트 예외로 가는 경우
- [ ] 이 규칙은 특정 업종/규제에만 해당하는가?
- [ ] 다른 프로젝트에서는 다르게 적용할 수 있는가?
- [ ] 제품 **브랜드/톤/도메인**에 따라 달라지는가?

---

## 예시: X2 프로젝트

### X2에서 전역 규칙 그대로 쓰는 것
- 해요체 ✓
- 한 화면 한 행동 ✓
- 카드형 선택 ✓
- empty state 3요소 ✓

### X2 프로젝트 예외
- 도메인 용어: "Intelligence Hub", "Signal Fusion" 등 → X2 전용 변환
- 업종 카드: 뷰티/F&B/금융/엔터 → X2 전용 설명
- 비교 타입: keyword_vs_keyword 등 → X2 전용 카드 문구
- 톤: 캐주얼 + 약한 재치 (업무 도구지만 딱딱하지 않게)
